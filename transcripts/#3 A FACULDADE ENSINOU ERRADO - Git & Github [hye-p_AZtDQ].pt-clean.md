# #3 A FACULDADE ENSINOU ERRADO: Git & GitHub

## Versionamento de código com Git

Git, adivinha só? É um binário que registra cada mudança no seu código, como se fosse um checkpoint de videogame. Não só nos dando o poder de voltar no checkpoint, como podemos comparar, mesclar e compartilhar checkpoints.

De forma visual, seria como se o Git criasse essa trilha de migalhas, onde cada migalhazinha dessa, na verdade, é um checkpoint. E digamos que você esteja aqui nesse checkpoint adicionando uma feature. Só que essa feature acaba quebrando a sua aplicação. Você pode simplesmente deletar esse checkpoint e voltar para esse checkpoint onde as coisas funcionavam.

## Por que Git é essencial em tempos de IA

Em tempos de desenvolvimento com IA, essa é a ferramenta mais importante de todas. Olha aqui você, meu querido macaquinho. Isso porque o output das IAs é imprevisível. Um prompt pode quebrar tudo. Portanto, a cada prompt, nós precisamos criar checkpoints para que você não fique nessa situação.

E a partir de agora, esses checkpoints serão chamados por nós de **commits**.

## Trabalho colaborativo

Além disso, como todos podem compartilhar e mesclar seus saves, o trabalho coletivo ficou muito mais fácil com Git.

Na analogia do videogame, seria como se, por exemplo, você e seu amigo estivessem tentando derrotar um chefão extremamente poderoso aqui no Dark Souls. Que então vocês pararam para refletir, chegaram à conclusão de que esse não é o único chefão poderoso. Existem vários chefões poderosos. Então, seria muito mais eficiente se vocês se separassem, cada um fosse derrotar um chefão diferente.

Então, o seu amigo foi derrotar o chefão X e criou um checkpoint. Você derrotou o chefão Y e gerou um checkpoint também. No final das contas, vocês fizeram o **merge**, ou seja, vocês mesclaram os dois checkpoints e criaram um novo checkpoint, onde tanto esse chefão foi derrotado, mas esse outro chefão também foi derrotado.

Trazendo para um exemplo real, seria como se seu amigo tivesse feito uma funcionalidade e criado um checkpoint, você fizesse outra funcionalidade e criasse o checkpoint. O Git, ele é responsável por fazer esse merge no código, gerando um novo código que tem tanto a funcionalidade que seu amigo fez quanto a funcionalidade que você fez.

Sim, meus amigos, o Git é mágico.

## A chegada do GitHub

E como se as coisas não pudessem ficar melhor, logo em seguida foi criado o **GitHub**, que basicamente é uma plataforma onde esses checkpoints podem ser salvos em nuvem e você e seu amigo do outro lado do mundo podem ter acesso a esse checkpoint, e o céu é o limite para o que vocês podem fazer juntos agora.

Conceitualizando, o GitHub é como um save em nuvem, onde os devs podem compartilhar seus códigos junto de seus checkpoints, commits, no que nós chamaremos de **repositório**.

## Deploy automático e rollback

Git, GitHub e uma série de outras funcionalidades começaram a ser derivadas do objetivo inicial do Git, como por exemplo, o **deploy automático**, ou seja, a hospedagem automática do seu aplicativo, que agora pode ser atualizado de forma automática baseado nos seus commits, garantindo que o servidor sempre vai estar auto atualizado com a última versão da sua aplicação.

E ainda por cima você tem a funcionalidade do **rollback**. Em caso de bug inserido, você pode simplesmente voltar para um commit e automaticamente o nosso app voltará para a versão funcional, pois justamente o deploy automático é uma funcionalidade onde plataformas de deploy ficam observando seus commits para sempre garantir que a sua aplicação sempre vai estar no último checkpoint que você considera pronto para o seu público.

## Open Source

Além disso, surgiu uma das melhores coisas do universo da programação, que é a iniciativa **Open Source**, onde devs programam de forma cooperativa repositórios com códigos responsáveis por lidar com determinado problema comum da comunidade. Então, a comunidade dos programadores é muito unida. Isso é muito massa.

Um exemplo incrível de projeto Open Source é o Linux. Foi criado por esse gostoso aqui, o **Linus Torvalds**. E caso você não saiba, o mesmo Linus Torvalds que tá... que tá... o DH bateu forte, hein? Que tá criando o Linux, também foi o cara, verdade, que criou o Linux. Pera aí que eu vou pegar minha Rita ali — é o mesmo cara que criou o Git. Então esse maluco aqui é incrível. Ele criou o Git, ele criou o Linux, ele é maravilhoso.

Então, ele criou o Linux e ele disponibilizou no GitHub para que qualquer pessoa pudesse sugerir melhorias, funcionalidades ou contribuir com o documento do projeto para facilitar a ingressão de novos devs no projeto. Além de que você também pode contribuir para a implementação de novas funcionalidades ao projeto.

## Forks

Pra você ter uma ideia do quanto o open source é incrível, existe uma coisa chamada **fork**, que basicamente é um processo no qual você pega um projeto de código livre e copia todo o código para dentro de um repositório que agora pertence a você.

Então temos, por exemplo, aqui o VS Code, motoqueiro F, que por muito tempo foi o editor de código mais querido e amado de toda a comunidade aí dos devs dinâmicos. Que com o surgimento da IA, surgiram vários projetos que justamente fizeram fork do código desse VS Code. Ou seja, eles pegaram o código desse VS Code, como ele era aberto, fizeram um clone em repositório próprio e adicionaram suas próprias funcionalidades, como é o caso do **Cursor**, que pegou o código do VS Code, adicionou inteligência artificial e agora eles estão comercializando essa solução, que é justamente o editor que nós utilizaremos para o Vibe Code, pois na minha opinião é o melhor editor no que diz respeito à integração com IA.

## Licenças

Agora é interessante de notar que nem todo projeto de código aberto você pode fazer isso, de simplesmente pegar o código, fazer um produto em cima e comercializar. Você precisa dar uma olhadinha se o criador do código sinaliza que ele autoriza você fazer isso.

Basicamente, ele vai deixar ali no repositório da aplicação uma informaçãozinha do tipo de licença a qual o projeto se qualifica. Se ele colocar **MIT**, por exemplo, significa que você pode modificar, você pode vender e você não necessariamente precisa abrir o código.

Mas não, meu querido, nem todo projeto Open Source é um projeto enorme como Linux, ou como esses editores de códigos incríveis. Um projeto open source pode ser uma simples lib responsável por facilitar alguma lógica. Esse tipo de projeto é muito mais fácil de manter e contribuir.
